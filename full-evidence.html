Class 00 - The HIMMA System: Building My Portfolio

For Class 00, my evidence is the HTML portfolio page itself. This was the first major step in organizing my work for CPIT-455. Instead of only submitting separate homework answers, I wanted to build a real portfolio that shows my weekly evidence, project work, and reflections in one place. The portfolio became more than just a container for assignments. It became a way to show my progress and the way I think throughout the course.

The main goal was to create a portfolio that feels like mine. I did not want it to look like a normal plain university page or a random template. I wanted it to reflect my style, so I chose a dark black theme with a clean and technical feeling. The background, typography, spacing, borders, and cards were all designed to feel serious, modern, and organized. The black style also made the portfolio feel different from a typical school document, which helped it feel more personal and professional at the same time.

One thing I focused on was making the page easy to navigate. I added a sidebar that stays visible so the user can move between sections without feeling lost. I also wanted the sidebar to show where the user currently is, so the active section lights up while scrolling. This small detail matters because it makes the portfolio feel more like a real interactive website, not just a long page of text.

I also worked on the scrolling experience. Instead of everything appearing flat, I wanted the sections to feel alive when the user moves through the page. The animations and transitions make the page feel smoother and more polished. They are not there just for decoration. They help guide the user's attention and make the portfolio easier to explore.

The structure of the portfolio was also important. I separated the content into clear sections such as About, Weekly Evidence, Project, and Reflection. For the weekly evidence, I used short summaries on the main page and longer answers in separate pages. This helped keep the main page clean while still allowing detailed work to be included. I learned that organization is part of design. A good portfolio should not overwhelm the reader, but it should still give enough depth when they want to see more.

Building the portfolio taught me a lot about frontend development and design decisions. I learned that small details can change how professional a website feels. Things like consistent spacing, readable text, hover effects, fixed navigation, section highlights, and image placement all matter. Even if each detail looks small by itself, together they make the page feel complete.

This task also showed me that software engineering is not only about writing code that works. It is also about building something usable, clear, and maintainable. A portfolio has users too. The user needs to understand where they are, what each section means, and how to access the evidence. That made me think more about user experience and the purpose behind each design choice.

Overall, Class 00 was about building the foundation for the rest of the course. The HTML portfolio became the place where I could show proof of learning, not just finished answers. I learned how to take my work and present it in a way that feels organized and personal. I also learned that my design choices matter because they represent how I want my work to be seen. The final portfolio reflects my style, my effort, and the way I want to approach software engineering: not just completing the task, but making it feel real and well thought out.

Goal: Build a trustworthy banking service that is secure, reliable, available, and resilient.

Selected System

The selected real-world system is a mobile banking application. This system allows customers to check their balance, transfer money, pay bills, manage accounts, and receive banking notifications. A banking system must be dependable because users trust it with their money, personal information, and important financial transactions.

1. Required Dependability Attributes

Availability: The mobile banking system must be available whenever users need it. Customers may need to access their accounts, transfer money, or pay bills at any time. If the system is unavailable, users may lose trust in the bank and may not be able to complete important transactions.

Reliability: The system must work correctly over time without failure. For example, when a user transfers money, the system should send the correct amount to the correct account. It should not crash, duplicate transactions, or show an incorrect balance.

Security: Security is very important because the system stores private customer information and financial data. The banking system must protect accounts from hacking, stolen passwords, phishing, unauthorized access, and fraud.

Safety: In this system, safety means avoiding serious financial harm to users. A mistake in the system should not cause money to be lost, sent to the wrong account, or accessed by the wrong person.

Resilience: The system should recover quickly after failures, attacks, or unexpected problems. If a server fails or a cyberattack happens, the bank should have backup systems and recovery plans to restore service quickly.

2. Risks and Failure Sources

Hardware Failure: Hardware problems can happen if servers, storage devices, or network equipment stop working. This can make the banking app slow or unavailable. Example: A bank server crashes, so users cannot log in or complete transactions.

Software Failure: Software failures can happen because of coding mistakes, logic errors, or weak testing. These bugs may cause wrong calculations, failed transactions, or app crashes. Example: A bug causes the app to show the wrong balance after a transaction.

Operational Failure: Operational failures happen because of human mistakes, poor procedures, or weak organization. For example, an employee may misconfigure the system during maintenance, causing the service to go offline. Example: A maintenance update is done incorrectly, and the banking app becomes unavailable for users.

Security Threats: Security risks include hacking, phishing, stolen login details, malware, or unauthorized access to accounts. Example: An attacker steals a customer's password and tries to transfer money from the account.

Network Failure: The system depends on internet connection and communication between the mobile app, bank servers, and payment networks. If the network fails, transactions may be delayed or interrupted. Example: A transfer is interrupted because of a network problem, and the user is unsure if the money was sent or not.

3. Proposed Dependability Strategy

Fault Prevention: The bank should use strong coding standards, secure design rules, and professional development practices. This helps prevent errors before the system is released.

Verification and Validation: The system should be tested regularly to make sure it works correctly. Testing should include login, money transfers, bill payments, balance updates, and error handling.

Fault Tolerance: The system should continue working even if one part fails. For example, backup servers can take over if the main server stops working.

System Protection: The banking system should use strong protection methods such as encryption, firewalls, multi-factor authentication, fraud detection, and secure login systems.

Backup and Recovery: The bank should keep regular backups of customer and transaction data. It should also have a disaster recovery plan so the service can return quickly after a failure or cyberattack.

Monitoring and Alerts: The system should be monitored all the time to detect unusual activity, failed transactions, server problems, or possible attacks. Users should also receive alerts for important actions such as login attempts and money transfers.

Clear Operational Processes: The bank should have clear procedures for updates, maintenance, employee access, and incident response. This reduces human mistakes and helps the organization respond faster when something goes wrong.

Conclusion

A mobile banking application is a critical real-world system because it handles money, personal data, and important financial services. The most important dependability attributes for this system are availability, reliability, security, safety, and resilience. The main risks come from hardware failure, software bugs, operational mistakes, security threats, and network problems. To improve dependability, the system should use testing, strong security, backup servers, monitoring, recovery plans, and clear operational processes.

CLASS 02 - Reliability Engineering

Selected System

The selected real-world system is a mobile banking application. This system is used by bank customers to check account balances, transfer money, pay bills, manage cards, receive notifications, and complete financial transactions online.

1. System Profile

The main users of the system are bank customers, bank employees, and system administrators. Customers use the mobile app directly from their phones, while bank employees and administrators manage accounts, monitor transactions, handle support requests, and maintain the system.

The usage pattern is transaction-heavy and almost continuous. Many users may access the app at the same time, especially during salary days, bill payment periods, holidays, or business hours. The system must handle many transactions without losing accuracy or trust.

For a banking system, the most important concern is not only that the app opens, but that every transaction is handled correctly. A small failure, such as showing the wrong balance or sending money twice, can cause serious financial problems.

2. Chosen Metric

The chosen main metric is ROCOF, which means Rate of Occurrence of Failures.

ROCOF is the best metric for this system because mobile banking is transaction-heavy and used continuously. The goal is to reduce how often failures happen during banking operations. For example, the system should minimize failed transfers, incorrect balance updates, login errors, duplicate payments, and transaction interruptions.

POFOD, which means Probability of Failure on Demand, is useful for systems that are used only when needed, such as emergency or safety-critical systems. It focuses on whether the system fails when a specific request is made. This can apply to some banking actions, but it is not the best main metric because mobile banking is used repeatedly and continuously.

AVAIL, which means Availability, is also important because users expect the app to be accessible whenever they need it. However, availability alone is not enough. A banking app may be available, but still unreliable if it gives wrong results or fails during transactions.

Because of this, ROCOF is the best main metric, while AVAIL can be used as a supporting metric. This matches the lecture idea that banking systems are high-transaction systems where ROCOF and integrity are especially important.

3. Proposed Strategy

The proposed strategy is redundancy with automatic failover.

The mobile banking system should not depend on only one main server. Instead, it should use multiple servers running in parallel. If the main server fails, another backup server should automatically take over. This helps keep the service running and reduces downtime.

This strategy improves availability because users can still access the app even if one server has a problem. It also supports reliability because the system can continue processing transactions instead of stopping completely.

The system should also use transaction logs to record every important action, such as transfers, payments, login attempts, and balance updates. If a failure happens, the bank can use these logs to check what happened and recover correctly.

At the code level, the system should validate all inputs before processing transactions. For example, it should check the account number, transfer amount, user identity, and account balance before completing a money transfer. This reduces the chance of incorrect transactions.

4. Diagram Explanation

The diagram shows the mobile banking system as a group of connected parts. The user interacts with the mobile banking app. The app connects to the application server. The server communicates with the database, authentication service, payment gateway, notification service, and external banks or payment networks.

The most critical part is the application server because it controls most of the banking logic. To improve dependability, this server should have a backup server. If one server fails, the backup server can continue the service.

Conclusion

A mobile banking system is a strong example of a dependable system because it must be reliable, available, secure, and accurate. Its usage pattern is transaction-heavy and continuous, so ROCOF is the best main reliability metric. AVAIL is also important, but it does not fully measure transaction correctness. The best strategy is to use redundancy with automatic failover, supported by transaction logs and input validation. This makes the banking system more dependable and helps protect users from service interruptions and financial errors.

Class 04 - Engineering for Security: The Security Assurance Case

Selected System

The selected system is a mobile banking application. The system allows customers to log in, check account balances, transfer money, pay bills, manage cards, and receive transaction notifications.

1. System Description

The mobile banking system includes several main parts:

  • Mobile banking app
  • Authentication service
  • Application server
  • Transaction processing service
  • Customer account database
  • Notification service
  • Security monitoring and audit logs

The user interacts with the mobile banking app. The app sends requests to the application server. The server checks the user identity through the authentication service, processes banking actions, updates the database, and sends notifications when important actions happen.

The most important assets in this system are:

  • Customer accounts
  • Passwords and login credentials
  • Personal information
  • Transaction records
  • Bank balance data
  • Payment and transfer services

Because the system handles money and private customer data, confidentiality, integrity, and availability are all important.

2. Threat Analysis

The main adversary is a cyber attacker who wants to gain unauthorized access to customer accounts or steal financial information.

Misuse Case 1: Steal Login Credentials. An attacker may use phishing or fake login pages to trick the customer into entering their username, password, or OTP.

Misuse Case 2: Unauthorized Money Transfer. An attacker may try to access a customer account and transfer money without permission.

Misuse Case 3: Brute Force Login Attack. An attacker may repeatedly try different passwords until they guess the correct one.

Misuse Case 4: Database Attack. An attacker may try to exploit a vulnerability in the application to access customer records or transaction data.

Misuse Case 5: Denial-of-Service Attack. An attacker may overload the system with requests so real customers cannot access the banking service.

3. Design Rationale

The chosen strategy is defense in depth. This means the system should not depend on one single security control. Instead, it should use multiple layers of protection.

The system should include:

  • Multi-factor authentication to protect login
  • Encryption to protect customer data
  • Input validation to prevent injection attacks
  • Account lockout after repeated failed login attempts
  • Role-based access control for employees
  • Transaction confirmation for high-risk transfers
  • Audit logs to track user and system activity
  • Monitoring tools to detect suspicious behavior
  • Backups and recovery plans in case of breach or failure

This layout is chosen because one security mechanism can fail. For example, if a password is stolen, multi-factor authentication can still block the attacker. If an attacker tries repeated login attempts, account lockout and monitoring can reduce the risk. If a breach happens, logs and backups help the bank investigate and recover.

4. V&V Results

Penetration Testing: Ethical hackers test the system by trying to find vulnerabilities before real attackers do.

Static Analysis: Security tools scan the source code to detect problems such as weak input validation, insecure code, or possible injection vulnerabilities.

Login Monitoring: The system records failed login attempts and detects suspicious patterns, such as many failed attempts from the same IP address.

Audit Logs: The system keeps records of important actions such as logins, password changes, money transfers, and employee access.

Encryption Testing: The system checks that sensitive data is encrypted during transmission and storage.

Backup and Recovery Testing: The bank tests whether customer and transaction data can be restored after a system failure or attack.

Conclusion

A mobile banking system needs a strong security assurance case because it handles money, private data, and important customer services. The main threats include stolen credentials, unauthorized transfers, brute force attacks, database attacks, and denial-of-service attacks. The best strategy is defense in depth, using several layers of protection instead of relying on one control. The security case is supported by evidence such as penetration tests, static analysis, monitoring logs, audit trails, and recovery tests.

CLASS 05 - Resilience Engineering

Class 05 - Resilience Engineering: Strengthening the Defense

Selected System

The selected system is a mobile banking application. This system allows customers to check balances, transfer money, pay bills, manage cards, and receive banking notifications. Since users depend on it for important financial services, the system must not only be reliable, but also resilient.

A resilient system is one that can continue delivering critical services during disruptive events and recover quickly afterward. In a banking system, this means that even if part of the system fails, users should still be able to access the most important services with minimal disruption.

1. Analyze the Failure Chain

Fault: A possible fault in the system is a bug in the transaction processing module. For example, under high load, the software may incorrectly update transaction status after a transfer request is submitted.

Error: The resulting error is an incorrect internal state. The transaction engine may record a transfer as completed in one part of the system, while the customer balance service or confirmation service still shows it as pending or failed.

Failure: The visible failure is what the user experiences. In this case, the user may see the transfer stuck in pending status, receive an incorrect confirmation, or see an incorrect account balance. This is a failure because the service delivered to the user no longer matches expected banking behavior.

So the failure chain is: fault, a bug in transaction processing logic; error, inconsistent transaction state between internal services; failure, the user sees a failed, pending, or incorrect transfer result.

2. Recognition and Resistance Strategies

Recognition Strategy: The system should use transaction integrity monitoring and log-based anomaly detection. If the monitoring system detects an unusual increase in pending transfers, mismatched account balances, or repeated transaction retries, it should immediately raise an alert.

This recognition approach matches the lecture idea that resilient systems must detect threats or failures early before they spread through the whole system.

Resistance Strategy: The system should switch to a backup transaction service and use replayable transaction logs. If the main transaction processor becomes unstable, a backup service can continue handling essential requests. At the same time, transaction logs can preserve all recent actions so incomplete operations can be safely replayed or recovered later.

This helps maintain critical service delivery during the event. Even if some advanced features are temporarily slowed down, the system can still preserve essential banking operations and reduce total outage.

EventRecognitionResistance
Transaction service inconsistency Transaction integrity checker, anomaly detection, automated log analysis, alerts to administrators Backup transaction service, replayable transaction logs, temporary queue, graceful degradation

3. Apply the Swiss Cheese Model

The Swiss Cheese Model says that system failure happens when weaknesses in different layers line up. To prevent this, the banking system should use multiple defensive layers, make them different, and reduce the holes in each layer.

Technical Layer: This includes system monitoring, transaction logs, backup server, database backup, integrity checker, and access controls. Possible holes include monitoring thresholds set too high, backup servers not updated properly, and delayed alert generation.

Human Layer: This includes support engineers, incident response staff, and operations teams who review alerts and respond to failures. Possible holes include delayed operator response, lack of training, and wrong recovery decisions under stress.

Organizational Layer: This includes incident response procedures, escalation policy, recovery planning, and communication processes. Possible holes include weak documentation, unclear incident ownership, and poor coordination between teams.

To make sure the holes do not align, the system should:

  • keep layers different, so one weakness does not affect all layers
  • test backup and recovery plans regularly
  • train staff on incident response
  • review monitoring rules and thresholds
  • clearly define escalation responsibilities
  • learn from previous failures and improve procedures

This follows the lecture idea that more defensive layers, more diversity, and fewer holes reduce the chance of catastrophic failure.

4. The Sociotechnical Factor

Human experts are important in resilience engineering because automation has limits. If the automated monitoring and backup systems cannot fully resolve the problem, a human expert must step in.

In this banking system, the human-in-the-loop would be a system administrator or incident response engineer. Their role would be to confirm the failure using logs and dashboards, decide whether to switch fully to the backup service, stop risky transaction processing if inconsistency is spreading, coordinate with database and support teams, communicate with users or customer service teams, and supervise safe restoration of the main service.

This matches the lecture's sociotechnical view that resilience depends on people, technology, and organization working together, and that human expertise remains essential.

Conclusion

The mobile banking system becomes resilient when it is designed not only to avoid failure, but also to detect problems early, resist disruption, and recover safely. In this case, the fault is a transaction processing bug, the error is an inconsistent internal state, and the visible failure is incorrect or delayed transaction results for users. The best resilience approach is to combine transaction monitoring, backup processing, replayable logs, layered defenses, and trained human experts. This allows the bank to continue delivering critical services even during unexpected failures.

CLASS 06 - Software Reuse Strategy

Class 06 - Software Reuse: Strategic Integration

Selected System

The selected system is an e-commerce store. This system allows customers to browse products, add items to a cart, create accounts, complete payments, track orders, and receive notifications about purchases and delivery.

1. Position the System in the Reuse Landscape

The chosen reuse approach is Application Framework with COTS/API integration.

An e-commerce store is a good example of software reuse because many parts of the system do not need to be built from zero. The development team can reuse an application framework for the website structure and connect it with ready-made services such as payment gateways, shipping APIs, notification services, and inventory modules.

This choice fits the reuse landscape because the system combines reusable framework features with external application services. Instead of writing every part manually, the team reuses tested components and focuses on connecting them correctly.

Development Schedule: Using reusable components helps reduce development time. Features like login, shopping cart, checkout, payment, and notifications are common in most e-commerce stores, so reusing them is faster than building them from scratch.

Team Skills: A small team may not have deep experience in payment security, inventory management, or delivery tracking. Reusing trusted APIs and frameworks allows the team to depend on specialized components while focusing on store-specific requirements.

Software Criticality: An e-commerce store is not safety-critical like a medical system, but it is still important because it handles payments, customer data, and orders. If reused components are not validated, they can cause payment errors, incorrect inventory, or failed orders.

Application Domain: E-commerce systems share many common features. This makes reuse suitable because the same structure can support different online stores with different products, branding, and business rules.

Execution Platform: The system is usually web-based and may also support mobile users. Reusable web frameworks and cloud services fit this platform well.

2. Define the Product Line Architecture

Core Components: These are the reusable parts that stay mostly the same across different e-commerce stores.

  • User registration and login
  • Product catalog
  • Product search and filtering
  • Shopping cart
  • Checkout process
  • Payment gateway integration
  • Inventory management
  • Order management
  • Order tracking
  • Notification service
  • Customer profile
  • Admin dashboard
  • Audit logs
  • Backup and recovery service

These components are generic because most e-commerce stores need them regardless of what products they sell.

Specialized Components: These are the parts that can be modified or configured for a specific store.

  • Store branding and design
  • Product categories
  • Discount and coupon rules
  • Delivery zones and shipping prices
  • Return and refund policy
  • Local payment methods
  • Tax settings
  • Loyalty points
  • Product recommendation rules
  • Seller or vendor rules
  • Language and currency settings
  • Customer support workflow

For example, a clothing store may need size and color filters, while an electronics store may need warranty options and technical specifications. The core e-commerce structure stays the same, but the specialized components change depending on the business.

3. Validation and the Ariane 5 Lesson

The Ariane 5 lesson shows that software reuse can be dangerous if reused components are not validated in the new context. A reused software module from Ariane 4 failed in Ariane 5 because the new system had different conditions, causing an overflow error.

For an e-commerce store, a reused payment gateway, shopping cart, or inventory module may work in one store but fail in another if the transaction volume, product rules, currency, or discount logic is different. Because of this, reused components must be tested before deployment.

Validation Plan:

  • Unit testing for reused modules such as cart, checkout, and inventory
  • Integration testing between the store, payment gateway, shipping API, and notification service
  • Boundary testing for prices, quantities, discounts, and payment limits
  • Security testing for login, checkout, and customer data
  • Load testing during high traffic periods such as sales or holidays
  • Failure testing for payment interruption or shipping API outage
  • Data consistency testing between orders, inventory, and payment confirmation
  • User acceptance testing to confirm the customer flow works correctly

Reliable Programming Commandment: Validate all inputs and check numeric limits.

This is important because e-commerce systems handle prices, quantities, discounts, card details, addresses, and payment amounts. If the system does not validate these values, it may accept invalid quantities, calculate the wrong total, apply discounts incorrectly, or create payment errors. Every reused component must check inputs before processing them.

4. Industry Alignment

The reuse strategy leverages microservices and cloud computing.

Microservices support reuse because each major feature can be separated into a service, such as product catalog, cart, payment, inventory, notification, and order tracking. This improves isolation because if one service fails, the whole store does not have to fail.

Cloud computing supports resilience by providing scalable servers, managed databases, backup services, monitoring, and recovery tools. This is useful during high demand, such as Black Friday or seasonal sales, because the system can scale and recover more easily.

Connection to Assignment 6 Resilience Goals: This reuse strategy supports resilience because reused services can be tested, monitored, replaced, and recovered separately. For example, if the notification service fails, customers can still place orders. If the payment gateway has an issue, the system can show a clear error, save the cart, and allow the customer to retry later. This prevents one reused component from causing a full system failure.

Conclusion

For the e-commerce store system, the best reuse strategy is Application Framework with COTS/API integration. The core reusable components include login, product catalog, cart, checkout, payment, inventory, order tracking, notifications, admin dashboard, and audit logs. The specialized components include branding, product categories, discounts, delivery rules, return policy, local payment methods, and currency settings. Reuse reduces development time and improves consistency, but it must be supported by careful validation. The Ariane 5 lesson shows that reused software should never be trusted blindly in a new context. By using testing, input validation, boundary checks, microservices, and cloud recovery, the system can support both software reuse and resilience.

Class 07 - CBSE Strategy: Component-Based Software Engineering

Selected System

The selected system is an e-commerce store. This system allows customers to browse products, create accounts, add items to a cart, complete payments, track orders, and receive notifications.

1. Position the System in CBSE

The chosen approach is Development with Reuse.

Development with reuse means building a new application using existing components and services instead of developing everything from scratch. This fits an e-commerce store because many required features already exist as reusable components, such as payment gateways, login systems, product catalog modules, shipping APIs, notification services, and analytics tools.

This choice is suitable because the goal is to reduce development time, improve reliability, and use off-the-shelf components where possible. In a university project or small team project, building every part from scratch would take too much time and could introduce more errors.

Budget: Using existing components reduces the cost because the team does not need to build every feature manually. For example, using a payment gateway API is cheaper and safer than creating a full payment system from zero.

Development Schedule: The schedule becomes shorter because reusable components such as authentication, payment, inventory, and notifications can be integrated quickly.

Market Availability: There are many available e-commerce components and APIs, such as payment gateways, shipping services, email/SMS services, and cloud storage. This makes development with reuse realistic.

Software Dependability: Some reused components are already tested in real systems. However, they still need validation in the new system because a component that works alone may fail when integrated with other components.

2. Component Composition

The system will be composed of several independent components.

Independent Components:

  • User Authentication Component
  • Product Catalog Component
  • Shopping Cart Component
  • Checkout Component
  • Payment Gateway Component
  • Inventory Management Component
  • Shipping and Tracking API
  • Notification Service
  • Order Management Component
  • Admin Dashboard Component
  • Audit Log Component

Interface Design:

Customer App / Website: Requires product list, login service, cart service, checkout service, and order tracking. Provides user requests, selected products, addresses, and payment details.

Authentication Component: Provides login, registration, password reset, and session validation. Requires user database access and security rules.

Product Catalog Component: Provides product details, prices, stock status, categories, and search results. Requires product database and inventory updates.

Shopping Cart Component: Provides add item, remove item, update quantity, and calculate subtotal. Requires product price and stock availability from the catalog and inventory components.

Checkout Component: Provides order confirmation, total price calculation, and checkout flow. Requires cart data, user address, payment service, and inventory confirmation.

Payment Gateway Component: Provides payment authorization and payment result. Requires payment amount, order ID, customer payment details, and secure connection.

Inventory Management Component: Provides stock availability and stock reservation. Requires order details after checkout.

Shipping and Tracking API: Provides delivery cost, delivery status, and tracking number. Requires customer address, order weight, and delivery option.

Notification Service: Provides email or SMS confirmation. Requires user contact details and order status.

Order Management Component: Provides order creation, order updates, cancellation, and history. Requires payment confirmation, inventory reservation, and shipping status.

Audit Log Component: Provides records of important actions such as login, payment, order update, and admin changes. Requires event data from all major components.

3. Validation and Risk Measurement

The system should be tested by checking how components work together, not only how each component works alone.

Testing plan:

  • Test login with the shopping cart and checkout
  • Test product stock updates after checkout
  • Test payment gateway response with order confirmation
  • Test failed payment handling
  • Test shipping API response with order tracking
  • Test notification service after successful and failed orders
  • Test audit logs for payment and admin actions
  • Test high traffic during sales or peak shopping periods
  • Test invalid input such as wrong address, negative quantity, or expired card

Potential Risk: A major risk is component mismatch. For example, the payment gateway may return payment status in a format that the order management component does not understand. This can cause the order to stay pending even though the customer was charged.

How to handle it: The system should use an adaptor component or glue code. The adaptor will translate the payment gateway response into the format required by the order management component. This matches the lecture idea that adaptor components help solve interface incompatibility between composed components.

Another risk is validation failure. A reused component may have hidden behavior or unwanted functionality. To reduce this risk, the team should create a test harness, review documentation, test edge cases, and check the component supplier's trustworthiness.

4. Industry Alignment

The modern industry trend that supports this component strategy is Composable AI and microservice-based architecture.

Composable AI means building systems from modular and interchangeable AI components. For an e-commerce store, this could include a reusable recommendation engine, chatbot support, fraud detection model, or review sentiment analysis component.

Microservices also support CBSE because each part of the system can be built, deployed, tested, and recovered separately. For example, if the notification service fails, the user can still complete an order. If the recommendation component fails, the product catalog can still work.

This improves system efficiency because each component can be updated or replaced without changing the entire system.

Conclusion

For the e-commerce store, the best CBSE approach is Development with Reuse. The system can be built by integrating independent components such as authentication, catalog, cart, checkout, payment, inventory, shipping, notifications, order management, and audit logs. Each component should have clear provides and requires interfaces so that communication is controlled and testable. The main risks are component mismatch, validation failure, and unreliable third-party services. These risks can be reduced through integration testing, adaptor components, test harnesses, and careful supplier selection. This strategy supports efficiency because it avoids unnecessary from-scratch coding while still keeping the system reliable and maintainable.

CLASS 08 - Distributed Architecture Plan

Assignment 8 Class 08 - Distributed Architecture Plan

Selected System

The selected system is an e-commerce store. This system allows customers to browse products, create accounts, add items to a cart, complete checkout, pay online, track orders, and receive notifications.

1. Position the System in the Distributed Landscape

The chosen model is a hybrid multi-tier client-server architecture with distributed backend components.

This means the system uses a clear multi-tier structure:

  • Client tier
  • API gateway / web server tier
  • Application service tier
  • Data storage tier

Inside the application service tier, the system uses distributed components such as product catalog, cart, checkout, payment, inventory, shipping, notification, and order management services.

This is the best model for an e-commerce store because the system must handle many users at the same time, especially during sales, holidays, or special offers. A single server would be risky because it could become overloaded or fail. By distributing the services across multiple nodes, the system can scale specific parts independently.

For example, during a large sale, the product catalog and cart services may receive more traffic than the notification service. With a distributed component architecture, the system can add more instances of the busy services without changing the whole system.

This model also supports resource sharing, fault tolerance, concurrency, and openness through standard protocols. These are important distributed system characteristics from the lecture.

2. Define the Distributed Architecture

Nodes / Devices:

1. Customer Device: This can be a browser or mobile app. The customer uses it to browse products, add items to the cart, and place orders.

2. CDN / Static Content Server: This stores static files such as images, CSS, JavaScript, and product images. It helps reduce latency by serving content from locations closer to the user.

3. Load Balancer / API Gateway: This receives user requests and routes them to the correct backend service. It also helps with traffic control, authentication checks, and rate limiting.

4. Product Catalog Service: This manages product details, categories, search, filtering, and product availability.

5. Shopping Cart Service: This manages the user's selected items, quantities, subtotal, and cart state.

6. Checkout and Order Service: This creates orders, confirms cart details, calculates totals, and coordinates payment and inventory updates.

7. Payment Service / Payment Gateway: This handles payment authorization and communicates with an external payment provider.

8. Inventory Service: This checks stock availability and reserves products after checkout.

9. Shipping and Tracking Service: This calculates shipping cost, creates tracking details, and connects to delivery providers.

10. Notification Service: This sends email or SMS notifications for order confirmation, payment status, shipping updates, and delivery alerts.

11. Database Cluster: This stores users, products, carts, orders, inventory, and payment status. It should use replication so data is not lost if one database node fails.

12. Monitoring and Logging Service: This records system events, errors, performance metrics, and security alerts.

How Data Transfer Happens with Low Latency:

  • The customer connects to the system through HTTPS.
  • Static files and product images are served through a CDN to reduce loading time.
  • The API gateway routes requests to the correct service.
  • Backend services communicate using REST APIs or message queues.
  • Frequently accessed product data is cached.
  • Database replicas are used to reduce read delays.
  • Message queues are used for non-immediate tasks such as notifications and shipping updates.

Example data flow:

Customer selects a product: Customer device -> CDN/product page -> API Gateway -> Product Catalog Service -> Database/cache -> Customer device.

Customer places an order: Customer device -> API Gateway -> Checkout Service -> Inventory Service -> Payment Service -> Order Database -> Notification Service.

3. Reliability and Security

Failover Plan:

The system should not crash completely if one node or service fails. The failover plan includes load balancer failover, service replication, database replication, queue-based recovery, graceful degradation, and backup/disaster recovery.

Load Balancer Failover: The load balancer checks the health of service instances. If one instance fails, traffic is redirected to another healthy instance.

Service Replication: Important services such as product catalog, cart, checkout, and payment should run in multiple instances. If one instance fails, another can continue handling requests.

Database Replication: The database should have a primary node and replica nodes. If the primary database fails, a replica can be promoted to continue service.

Queue-Based Recovery: If the notification service or shipping API is unavailable, order events can be stored in a message queue and processed later. This prevents the whole checkout process from failing because of a secondary service.

Graceful Degradation: If a non-critical service fails, the system should continue essential services. For example, if the recommendation service fails, customers can still browse products and complete checkout.

Backup and Disaster Recovery: The system should take regular backups of important data such as orders, users, payments, and inventory. Backups should be tested regularly to make sure recovery works.

Security Standard / Protocol: The system should use HTTPS with TLS to protect data in transit.

This is important because customer information, login credentials, addresses, and payment-related data move between the client, gateway, and backend services. TLS encrypts the connection so attackers cannot easily read or modify the data while it is moving across the network.

Other security controls include:

  • Authentication tokens for user sessions
  • API gateway rate limiting
  • Firewall rules
  • Input validation
  • Access control between internal services
  • Secure logging without exposing sensitive payment data
  • Monitoring for unusual traffic or failed login attempts

The lecture lists security risks in distributed systems such as interception, interruption, modification, and fabrication. These controls help reduce those risks.

4. Industry Alignment

This strategy uses cloud computing and SaaS.

Cloud computing supports the e-commerce system because it provides on-demand resources. During normal traffic, the store can run with fewer resources. During peak sales, the cloud can automatically add more service instances to handle more users. This reduces the need to buy and maintain expensive infrastructure for the highest possible traffic all year.

SaaS and cloud services can also be used for parts of the system, such as email notifications, payment processing, analytics, monitoring, and customer support. This reduces maintenance work and allows the development team to focus on the main e-commerce system.

The lecture explains that SaaS allows applications to be accessed over the internet without users managing complex software or hardware, and that it is scalable, cost-effective, and low maintenance.

Connection to Continuity and Resilience:

This distributed architecture supports continuity because the system can keep running even when one part fails. It also supports resilience because services can be restarted, replaced, scaled, or recovered independently.

  • If one product catalog server fails, another instance continues.
  • If the payment gateway is slow, the checkout service can show a retry message and protect the cart.
  • If the notification service fails, order confirmation can be queued and sent later.
  • If a database node fails, a replica can continue service.
  • If traffic increases, cloud auto-scaling can add more resources.

Conclusion

For the e-commerce store, the best distributed architecture is a hybrid multi-tier client-server model with distributed backend components. This design separates the client, API gateway, application services, and data layer. It supports scalability by allowing busy services to scale independently, improves reliability through failover and replication, and protects data using HTTPS/TLS. Cloud computing and SaaS also improve the system by providing on-demand resources, lower infrastructure cost, and easier recovery. This architecture supports the real-world goals of continuity and resilience because the system can continue operating even during outages, traffic spikes, or partial service failures.

CLASS 09 - Multi-tier and Middleware Strategy

Class 09 - Multi-tier and Middleware Strategy

Selected System

The selected system is an e-commerce store. This system allows users to browse products, add items to a cart, complete checkout, pay online, track orders, and receive notifications.

1. Tier Architecture Selection

The chosen architecture is a multi-tier architecture.

I chose multi-tier architecture instead of thin-client or fat-client because an e-commerce system has many parts that should be separated. If everything is placed on one server or one client, the system becomes harder to scale, maintain, and recover from failures.

A thin-client approach would make the client simple, but it may overload the server because most processing happens on the server side. A fat-client approach would move more processing to the user device, but it would make updates, security, and client management more difficult.

Multi-tier architecture is better because it separates the system into clear layers. The user interface, business logic, and database are not mixed together. This makes the system easier to maintain and allows each tier to scale independently.

For an e-commerce store, this is important because the system may experience high traffic during sales, holidays, or product launches. The application tier can be scaled by adding more service instances, while the database tier can use replication and backups to protect data.

2. Define the Multi-Tier Architecture

Presentation Tier: The presentation tier is the client side of the system. It includes the customer browser or mobile app.

Main role:

  • display product pages
  • allow users to search and filter products
  • manage the shopping cart interface
  • collect login, address, and checkout information
  • show order status and notifications

This tier should focus mainly on user interaction, not heavy business logic.

Application Tier: The application tier is the server side of the system. It contains the main business logic and services.

Main components:

  • API Gateway
  • Authentication Service
  • Product Catalog Service
  • Shopping Cart Service
  • Checkout and Order Service
  • Payment Service
  • Inventory Service
  • Shipping Service
  • Notification Service
  • Monitoring and Logging Service

Main role:

  • process user requests
  • check authentication
  • calculate order totals
  • validate stock availability
  • communicate with payment and shipping services
  • send notifications
  • enforce business rules

Data Tier: The data tier stores and protects the system data.

Main components:

  • User Database
  • Product Database
  • Order Database
  • Inventory Database
  • Payment Status Records
  • Cache
  • Backup Storage
  • Audit Logs

Main role:

  • store customer accounts
  • store product and inventory data
  • store orders and payment status
  • support recovery using backups
  • improve performance using cache

3. Role of Middleware

Middleware is the layer that helps the different system parts communicate. In this e-commerce system, middleware connects the presentation tier, application services, databases, and external APIs.

Middleware examples:

  • API Gateway
  • Database middleware
  • Message queue
  • Service adapter
  • Authentication middleware
  • Logging middleware

The API Gateway receives requests from the client and routes them to the correct service. For example, a product search request goes to the Product Catalog Service, while a payment request goes to the Checkout and Payment Services.

Database middleware helps application services communicate with databases. It handles connection management, query formatting, data transfer, and response handling. Middleware allows a web server to communicate with a database server and simplifies database interaction.

Message queues are also useful because they allow non-urgent tasks to happen asynchronously. For example, after an order is created, the notification service can send an email or SMS without slowing down the checkout process.

4. Validation and Localized Control

To make sure a failure in one tier does not collapse the whole system, the design should include localized control and failure handling.

If the Database Tier Fails: The application tier should not silently fail. Instead, it should detect the database problem and show a clear message to the user, such as "Order service is temporarily unavailable, please try again shortly."

The system should also:

  • use database replicas
  • use backups
  • use cached product data for browsing
  • stop checkout temporarily if order storage is unavailable
  • keep logs of failed requests
  • alert administrators immediately

If the Application Tier Fails: The load balancer should redirect traffic to another healthy application server. If only one service fails, the rest of the system should continue working.

Example: If the notification service fails, customers should still be able to place orders. The message can be queued and sent later.

If the Presentation Tier Fails: If the user interface has an issue, the backend services should remain protected. The client should not directly access the database or payment systems.

Failure Prevention and Testing:

  • test database outage scenarios
  • test payment gateway failure
  • test slow network conditions
  • test service timeout handling
  • test message queue recovery
  • test backup restoration
  • monitor errors in each tier

This prevents silent failure because every tier has clear responsibilities, error handling, logs, and fallback behavior.

5. Industry Alignment

The modern trend that aligns with this design is microservices architecture.

Microservices fit multi-tier architecture because the application tier can be divided into independent services. Each service can be scaled, updated, and recovered separately.

For example:

  • Product Catalog Service can scale during browsing traffic
  • Checkout Service can scale during sales
  • Notification Service can fail without stopping payment
  • Payment Service can be isolated for security
  • Inventory Service can be updated without changing the UI

This supports dynamic scaling and independent service updates. It also makes the system easier to maintain because each service has a clear role.

Conclusion

For the e-commerce store, a multi-tier architecture is the best choice because it separates the presentation tier, application tier, and data tier. This reduces server overload, improves scalability, and makes the system easier to maintain. Middleware such as API gateways, database middleware, service adapters, and message queues connects the distributed components and helps them communicate safely. If one tier fails, the system can detect the issue, isolate the failure, alert administrators, and continue partial service where possible. This supports easier maintenance, better resilience, and stronger future adaptability.
